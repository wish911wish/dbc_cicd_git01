## 1. 目的・範囲

### 1.1 目的

複数のテーブル・層・ジョブを結合した状態で、データが設計どおりに流れ、整合することを検証する。単体で担保した変換ロジックを結合し、ストレージ層の挙動・履歴管理・オーケストレーションまで含めた振る舞いを確認する。

### 1.2 結合テストの対象

| 対象 | 内容 |
|------|------|
| 層間連携 | Bronze → Silver → Gold のデータフロー整合 |
| 複数テーブル整合 | Hub/Link/Satellite 間の参照整合、Goldマートの集計整合 |
| ストレージ挙動 | Delta MERGE の挙動（存在時スキップ/変更時INSERT等）、冪等性 |
| 履歴・CDC | AUTO CDC による SCD Type 2 の履歴遷移、有効期間・currentフラグ |
| PIT / Bridge | 参照時点のSatellite整合、基準時刻の網羅 |
| パイプライン | Lakeflow Declarative Pipelines のE2E実行、Expectations/quarantineの全体挙動 |
| オーケストレーション | Lakeflow Jobs のタスク依存・パラメータ受け渡し・リトライ |
| デプロイ整合 | DAB（targets/変数）でデプロイした資産が想定どおり動作 |

### 1.3 対象外

個々の変換関数・単一ルールの正当性は単体テストで担保済みとして、本書では再検証しない。非機能（性能・障害回復・SLA）・移行データ全量検証はシステムテスト/移行テストで扱う。

## 2. 結合テストの観点

### 2.1 データフロー・整合観点

| 観点 | 確認内容 |
|------|----------|
| 件数整合 | 入力 ⇔ 中間 ⇔ 出力 ⇔ quarantine の件数が説明可能 |
| 参照整合 | Link の参照先Hubが存在、孤立キーがない |
| 集計整合 | Goldの集計値が元データと一致 |
| 一意性 | 業務キー・ハッシュキーの一意性が層をまたいでも保たれる |

### 2.2 ストレージ挙動・冪等性観点

| 観点 | 確認内容 |
|------|----------|
| Delta MERGE | Hub: 既存キーはスキップ／新規のみ追加。Satellite: hash diff一致時は追加しない |
| 冪等性 | 同一入力でパイプラインを再実行しても結果が変わらない |
| 増分処理 | 増分ロード時に既存データを破壊せず、差分のみ反映 |

### 2.3 履歴・CDC観点（AUTO CDC / SCD Type 2）

| 観点 | 確認内容 |
|------|----------|
| 履歴生成 | 属性変更時に新バージョンが生成される |
| 有効期間 | 開始/終了の連続性、期間の重複・欠落がない |
| current 一意性 | 1業務キーにつき current レコードが1件 |
| 遅延・順序 | 遅延到着・順序逆転データでも履歴が正しく構成される |
| 論理削除 | 削除フラグ/到達不能データの扱いが設計どおり |

### 2.4 PIT / Bridge 観点

| 観点 | 確認内容 |
|------|----------|
| 時点整合 | PIT経由で参照したSatellite値が、当該時点の正しい値と一致 |
| 基準時刻網羅 | PITの基準時刻が必要な粒度を網羅 |

### 2.5 パイプライン・オーケストレーション観点

| 観点 | 確認内容 |
|------|----------|
| E2E実行 | パイプラインが依存順に完走する |
| Expectations全体 | warn/drop/fail が全体フローで設計どおり作用、quarantineへ退避 |
| ジョブ依存 | Lakeflow Jobs のタスク依存・分岐・リトライが正しい |
| パラメータ | ジョブ/パイプラインへのパラメータ受け渡しが正しく反映 |
| 失敗時挙動 | タスク失敗時の停止/再実行が設計どおり |

## 3. テスト設計・データ

### 3.1 設計方針

業務シナリオ単位でテストケースを設計する。1シナリオで「初回ロード → 増分ロード（変更あり/なし）→ 再実行」のように、履歴と冪等性を時系列で検証できる流れを基本形とする。

### 3.2 テストデータ方針

| 種別 | 方針 |
|------|------|
| 合成シナリオデータ | 新規・変更・削除・遅延到着を含む時系列データを fixtures として用意 |
| 期待結果データ | 各ステップ後の期待状態（件数・履歴・current）を定義 |
| 本番相当データ | 結合テストでは使用しない。検証は合成シナリオデータのみで行う |

> 結合テストは本番相当データ（マスキング済みを含む）を使用しない。期待結果を厳密に定義でき、再現性と機密リスク低減の両面で有利な合成データに統一する。本番相当データを用いた傾向確認が必要な場合は、システムテスト等の後工程で扱う。

### 3.3 検証方法

実行後のテーブル状態を期待結果と突合する。件数・キー一意性・履歴整合・参照整合をアサーションで検証する。

## 4. 実行方式（pytest + DAB + Lakeflow Jobs）

### 4.1 構成・実行環境

| 項目 | 方針 |
|------|------|
| フレームワーク | pytest（結合シナリオの駆動・アサーション） |
| デプロイ | DAB で test 用 target に資産（Pipeline/Job）をデプロイ |
| 実行 | Lakeflow Jobs / パイプラインを起動し、完了後に結果を検証 |
| 実行契機 | {対象ブランチ / 手動承認} 契機で staging 相当環境にて実行（全体方針書 6章） |

### 4.2 スキーマ分離・teardown パターン

| 項目 | 方針 |
|------|------|
| スキーマ命名 | 実行ごとに一意のスキーマを生成（例: `it_{ブランチ}_{実行ID}`） |
| 分離単位 | テストスイート/シナリオ単位で他テストと干渉しないよう分離 |
| setup | fixture でスキーマ・必要テーブル・初期データを準備 |
| teardown | fixture スコープで確実に削除。テスト失敗時も残さない（finalizer / yield後処理） |
| 冪等な後始末 | teardown 自体が失敗しても他テストに影響しないよう冪等に実装 |

### 4.3 pytest 運用ルール

| 項目 | 方針 |
|------|------|
| marker | `@pytest.mark.integration` を付与し単体と分離。重い実行は `@pytest.mark.slow` 等で選別実行可能に |
| parametrize | シナリオ/データセットのバリエーションを parametrize で表現 |
| パラメータ受け渡し | テスト→ジョブ/パイプラインのパラメータ（対象スキーマ等）を明示的に注入 |
| fixture スコープ | スキーマ等の重い資源は適切なスコープ（session/module）で共有し、実行時間を抑制 |

### 4.4 証跡の取得・保存

テスト用 Job の実行結果（Lakeflow Jobs の Run 完了画面）を結合テストの証跡とする。

| 項目 | 方針 |
|------|------|
| 取得対象 | シナリオごとの Job Run 完了画面（結果ステータス・Run ID・実行日時） |
| 記録 | Run ID・結果・実行日時・対象target/スキーマを結果報告書に転記（リンクは補助） |
| 保存 | Run 画面のスクリーンショットを `evidence/` 配下に保存し版管理（リンク切れ・権限変更への対策） |
| 採番 | テストIDをシナリオ消化表・観点表と同一採番とし、相互参照可能にする |

> 証跡の具体的なレイアウトは結果報告書 2.3 を参照。タスク単位の証跡が必要な場合は task run 画面も取得する。

## 5. 合格基準・完了基準

> 全体方針書 4.3 の共通定義に対し、結合レベルのしきい値を設定する。

| 基準 | 結合での設定（例。プロジェクトで確定） |
|------|----------------------------------------|
| ケース消化率 | 計画シナリオの 100% を実施 |
| テスト成功率 | 100%（不合格は是正・再試験で解消） |
| データ品質 | Expectations fail 0件、quarantine件数が許容範囲内 |
| 履歴整合 | SCD2 の有効期間重複/欠落 0件、current一意性違反 0件 |
| 参照整合 | 孤立キー 0件 |
| 冪等性 | 再実行で差分が出ないことを確認 |
| 不具合 | 重要度「高」以上の未解決 0件、収束傾向を確認 |
