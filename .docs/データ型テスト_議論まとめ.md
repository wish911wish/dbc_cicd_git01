# データ型テスト 議論まとめ（引き継ぎ用）

> Databricks ETL プロジェクトにおける「データ型に着目したテスト」の検討内容を、別チャットで継続できるようまとめた引き継ぎ文書。決定事項・設計・サンプル・未決事項を含む。

---

## 0. 前提コンテキスト

- 対象：Databricks 上の ETL（メダリオン構成、Data Vault 2.0、Lakeflow Declarative Pipelines、Lakeflow Jobs、DAB、Unity Catalog）。
- 既に作成済みのテスト文書セット（全体方針書 / 単体・結合方針書 / 単体・結合結果報告書）があり、本議論はそのうち「データ型変換」のテスト設計を深掘りしたもの。
- 関連する既存の決定：
  - 単体テストはインメモリ実行・スキーマ不使用（書込前ロジックまで）。ストレージ挙動は結合テスト。
  - 結合テストは合成データのみ（本番相当データ不使用）。
  - 不正値は Expectations で quarantine へ退避する設計。

---

## 1. Databricks のデータ型（出典の要点）

公式の SQL データ型一覧に基づく分類：整数（TINYINT/SMALLINT/INT/BIGINT）、厳密数値（DECIMAL）、浮動小数点（FLOAT/DOUBLE）、日時（DATE/TIME/TIMESTAMP/TIMESTAMP_NTZ）、INTERVAL、STRING/BINARY/BOOLEAN、地理空間（GEOGRAPHY/GEOMETRY）、複合型（ARRAY/MAP/STRUCT/VARIANT/OBJECT）、untyped NULL（VOID）。

型クラス共通のテスト観点：境界値、NULL許容、型変換・キャスト、丸め・精度、特殊値（NaN/Infinity/-0.0）。

---

## 2. 最重要の前提：キャスト挙動（ANSIモード）

データ型テストの「期待値」は、変換方式（CAST / try_cast / to_date）と ANSI モードで変わる。これを先に固定しないとテストが書けない。

- 新しめの Databricks 環境では ANSI モードが既定で有効（2022-10-19 以降のアカウントは既定 TRUE、Spark 4.0 / DBR 17.0 以降は既定有効）。
- ANSI 有効下では、不正文字列の `CAST` は例外で Job を停止させる。`try_cast` は例外の代わりに NULL を返す。
- 注意点：文字列 `"1.0"` や `"1e0"` は INT への CAST で弾かれる（小数点・指数表記の文字列は INT 不可）。

| 入力例 | CAST(ANSI有効) | CAST(ANSI無効) | try_cast |
|--------|----------------|----------------|----------|
| "123"→INT | 123 | 123 | 123 |
| "abc"→INT | エラー | NULL | NULL |
| 範囲外→INT | エラー | NULL/不正 | NULL |
| "2026-13-01"→DATE | エラー | NULL | NULL |

---

## 3. 採用した設計方針

| 項目 | 決定 |
|------|------|
| 変換方式 | `try_cast`（ANSI下でも停止させず、失敗を NULL として検出） |
| 不正値の扱い | Expectations 相当の判定で quarantine テーブルへ退避（drop ではなく退避） |
| 有効性判定 | 「ソース非NULL かつ 変換結果NULL」＝変換失敗＝無効。ソースNULLは正規の欠損として有効（NULL→NULL） |
| シナリオ分割 | 「正常データのみ（完走確認）」と「異常データ含む（振り分け確認）」を別シナリオ |
| 件数整合 | 入力件数 = 本体件数 + quarantine 件数 を独立に検証 |

## 4. 型別テスト観点とサンプル値（リファレンス）

### STRING→INT
正常（"0"/"123"/"-456"）、境界（"2147483647"/"-2147483648"）、overflow（"2147483648"）、小数・指数文字列（"123.45"/"1.0"/"1e3"）、非数値（"abc"）、桁区切り/符号/前ゼロ（"1,234"/"+123"/"007"）、全角（"１２３"）、前後空白（" 123 "）、空・NULL。

### STRING→DATE
正常 ISO（"2026-06-19"）、時刻付き→DATE切詰め（"2026-06-19 09:30:00"）、別形式（"2026/06/19"/"20260619"/"06/19/2026"）、不正月日（"2026-13-01"/"2026-02-30"）、うるう年（"2024-02-29"有効/"2025-02-29"無効）、境界（"0001-01-01"/"9999-12-31"）、全角/前後空白、空・NULL。

### STRING→DECIMAL(p,s)
通常、scale 上限、丸め（"12.999"→"13.00" は**有効**）、精度超過（→NULL=無効）、負値・ゼロ・NULL。丸めは silent で無効ではない点に注意。

### STRING→STRING（低リスク）
通常、空文字、NULL、前後空白（トリム有無）、マルチバイト/絵文字/制御文字、最大長。加工があるときのみ重点的に。

## 5. 「全Jobで型テストをやるべきか」の結論

一律不要。「変換エンジンの挙動」と「Jobへの組み込み」を分けて考える。

| 検証対象 | 場所 | 全Jobで繰り返すか |
|----------|------|-------------------|
| try_cast/cast の挙動（"abc"→NULL 等） | 共通部品の単体テスト or 代表1Jobで一度 | No |
| 妥当性ルール（Expectations）の内容 | 各Job | Yes |
| 違反行の振り分け（quarantine） | 各Job | Yes |
| 件数整合（入力=本体+quarantine） | 各Job | Yes |
| 対象カラム・業務的許容値 | 各Job | Yes |

前提：変換を共通部品化し、各Jobはそれを呼ぶ構造であること。変換方式（try_cast/cast）と Expectations 設計が Job 間で同一であること。実装がバラバラなら各Jobで確認が要る → 共通部品化を先に検討。

---

## 6. 共通化の判断

- 共通化する価値が高い：定型キャストが複数Jobで繰り返される／品質ルールを統一したい／仕様変更に1箇所で追従したい（特に Bronze→Silver の一括キャスト）。
- 共通化しない：業務ロジック（マスタ変換・税計算・コード変換など）が絡む変換。型変換と業務変換は層を分ける。
- 早すぎる共通化は避ける。2〜3 Job で同型キャストの繰り返しが見えてから導入。
- 推奨粒度：単一カラム変換 / 妥当性判定（理由列付与）/ 振り分け。Job全体テンプレ化は過度。

---

## 7. 確定した実装パターン

### 変換Job（Lakeflow Declarative Pipelines）の3段構成
1. 型付け中間ビュー：生文字列 + `try_cast` 結果（`{col}_t`）+ `quarantine_reason`（理由連結）+ `_is_valid` フラグ。
2. 本テーブル：`_is_valid` の行のみ、型付け列を採用。
3. quarantine テーブル：`NOT _is_valid` の行を生文字列のまま + 理由 + 退避時刻で退避。

### 共通部品（業務要件を含まない）
- `cast_columns(df, mapping)`：`{カラム名: 目標型}` を受け取り、型付け列・quarantine_reason・_is_valid を付与。`try_cast` 使用。許可型のみ受理（誤用防止）。
- `split_valid_quarantine(df)`：`_is_valid` で有効/無効の2 DataFrame に分割。
- 業務ルール（正値のみ等）は共通部品に入れず、各Jobの Expectations で表現。

### テスト分担
- 型変換の挙動網羅（異常系フルセット）＝共通部品の単体テストで一度（pytest・parametrize・インメモリ）。
- 各Jobの結合テスト＝マッピングの正しさ・振り分け・件数整合を代表データ（有効1〜2件＋無効1〜2件）で確認。

## 9. 未決事項・次の論点（別チャットで継続）

- 非ISO形式日付（"2026/06/19" など）を業務的に受け入れるか。受け入れるなら `to_date(..., format)` を共通部品に format 引数で足すか、Job側で前処理するか。
- DECIMAL の丸め（silent）を許容とするか、桁落ちを検知・警告したいか。
- t_b 側に他型（TIMESTAMP/TIMESTAMP_NTZ、BOOLEAN、複合型 ARRAY/MAP/STRUCT、VARIANT）が増えた場合の観点追加。
- TIMESTAMP と TIMESTAMP_NTZ のタイムゾーン挙動テスト（セッションTZ依存 vs 非依存）。
- 共通部品の単体テストにおける DECIMAL 値の厳密比較方法（文字列化 vs Decimal型比較）。
- 共通化導入のトリガー（何Jobから部品化するか）の運用ルール化。
- Expectations メトリクスを `event_log()` で取得し、quarantine 件数の収束を可視化する仕組み。
