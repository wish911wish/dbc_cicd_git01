## 1. はじめに

### 1.1 目的

{プロジェクト名} のテスト工程全体における目的・範囲・方針・共通基準・体制を定義し、関係者間で品質保証の進め方を合意する。各テストレベルの実施方針は本書を起点に詳細化する。

### 1.2 適用範囲

Databricks 上の ETL／データパイプラインおよび関連成果物に対するテスト工程全体に適用する。対象アーキテクチャはメダリオン（Bronze/Silver/Gold）、Data Vault 2.0、Lakeflow Declarative Pipelines、Lakeflow Jobs、DAB、Unity Catalog を前提とする。

### 1.3 関連文書

| 区分 | 文書名 |
|------|--------|
| 上位 | 要件定義書 / アーキテクチャ設計書 / データモデル設計書（Data Vault 2.0） / ETL設計書 |
| 本書配下 | 単体テスト方針書 / 結合テスト方針書 |
| 本書配下 | 単体テスト結果報告書 / 結合テスト結果報告書 |

### 1.4 用語定義

| 用語 | 定義 |
|------|------|
| メダリオン | Bronze（生データ取込）／ Silver（整形・統合）／ Gold（業務利用）の3層構成 |
| Data Vault 2.0 | Hub（業務キー）／ Link（関連）／ Satellite（履歴属性）を中心としたモデリング手法 |
| PIT テーブル | Point-In-Time テーブル。特定時点のSatellite整合参照を高速化する補助構造 |
| Lakeflow Declarative Pipelines | 宣言的にテーブル依存とデータ品質を定義するパイプライン（旧 DLT） |
| Expectations | パイプライン内のデータ品質制約（warn / drop / fail） |
| AUTO CDC | 変更データ取込によるSCD Type 1/2 適用機能（旧 APPLY CHANGES） |
| DAB | Databricks Asset Bundles。資産（Job/Pipeline等）をコードで定義・デプロイ |
| Unity Catalog | catalog.schema.table の3階層名前空間によるガバナンス基盤 |

> 用語の詳細・補足は各レベル方針書側で必要に応じて追記する。

## 2. テスト対象システム概要

### 2.1 アーキテクチャ概要

```
[ソース] → Bronze（生データ） → Silver（整形/統合・Data Vault 2.0） → Gold（業務マート）
   変換+品質: Lakeflow Declarative Pipelines（Expectations）/ 履歴: AUTO CDC
   実行: Lakeflow Jobs / 定義・デプロイ: DAB（dev/staging/prod）/ ガバナンス: Unity Catalog
```

### 2.2 テスト対象範囲・対象外

| 区分 | 内容 |
|------|------|
| 対象 | メダリオン各層の変換、Data Vault 2.0 ロード、Expectations/quarantine、AUTO CDC、Lakeflow Jobs、DAB デプロイ、増分・冪等性 |
| 対象外 | Databricks プラットフォーム機能、ソース内部ロジック、{下流BI等} |

> 範囲の細目（どのテーブル/ジョブを各レベルで検証するか）は各レベル方針書で具体化する。プロジェクト固有の除外項目は必ず追記すること。

## 3. テスト全体方針

### 3.1 品質目標

| 観点 | 目標（例。プロジェクトで確定） |
|------|--------------------------------|
| 機能正当性 | 設計どおりの変換結果 |
| データ品質 | Expectations 違反が完了基準内に収束 |
| 履歴正当性 | SCD2 / Satellite 履歴の時系列整合 |
| 冪等性 | 再実行で結果が変わらない |
| 自動化率 | 単体・結合の自動実行率 {例: 90%} 以上 |

### 3.2 テストレベルの定義

| レベル | 主な確認内容 | 主担当 | 詳細方針 |
|--------|--------------|--------|----------|
| 単体 | 個々の変換関数・SQL・Expectationルール・単一テーブルのロードロジック | 開発者 | 単体テスト方針書 |
| 結合 | 複数テーブル/層をまたぐ整合性、E2Eパイプライン、CDC/PIT整合、依存・オーケストレーション | 開発/テスト担当 | 結合テスト方針書 |
| システム | 非機能（性能・運用・障害回復）、全体フロー、SLA | {担当} | {別途} |
| 受入 | 業務要件充足、利用者観点の妥当性 | 業務/受入担当 | {別途} |

### 3.3 テスト種別とレベルのマッピング

| 種別＼レベル | 単体 | 結合 | システム |
|--------------|:----:|:----:|:--------:|
| 機能 | ◎ | ◎ | ○ |
| データ品質 | ◎ | ◎ | ○ |
| 履歴・CDC | △ | ◎ | ○ |
| 回帰 | ◎ | ◎ | ○ |
| 移行 | − | ○ | ◎ |
| 性能 | − | △ | ◎ |
| セキュリティ | − | △ | ◎ |

凡例：◎=主実施 / ○=実施 / △=一部実施 / −=対象外

> 各種別の具体的な検証観点は、実施レベルに対応する方針書で定義する。

## 4. 共通基準・標準

> 本章は全テストレベルに共通する基準の「定義」を置く。各レベル方針書はこれを参照し、レベル固有のしきい値・観点を設定する。

### 4.1 テスト設計の共通方針

同値分割・境界値分析・デシジョンテーブルを基本とする。データパイプライン共通の検証軸として、件数整合・NULL/重複/型・キー一意性・増分/冪等性・時系列整合を標準とする。各観点の詳細な適用（対象テーブル・データパターン）は各レベル方針書で具体化する。

### 4.2 データ品質検証の共通観点

Data Vault / メダリオン構造に対する検証は、以下を共通の軸とする。詳細な確認項目は各レベル方針書に展開する。

| 対象 | 共通の検証軸 | 主な実施レベル |
|------|--------------|----------------|
| Hub / Link / Satellite | キー一意性・ハッシュ生成規則・履歴整合 | 単体（単一テーブル）/ 結合（参照整合） |
| PIT / Bridge | 参照時点の整合 | 結合 |
| AUTO CDC (SCD2) | 有効期間の連続性・current一意性 | 結合 |
| Expectations | warn/drop/fail 動作・quarantine | 単体（ルール）/ 結合（全体挙動） |

### 4.3 合格基準・完了基準（共通定義）

各レベル方針書はこれを引用し、レベル固有のしきい値を設定する。

| 基準 | 共通定義（しきい値はレベル方針書で設定） |
|------|------------------------------------------|
| ケース消化率 | 計画ケースの所定割合を実施 |
| テスト成功率 | 不合格は是正・再試験で解消 |
| 不具合収束 | 重要度「高」以上の未解決0件、収束傾向を確認 |
| カバレッジ | ロジックカバレッジを測定（目標値はレベルで設定） |
| データ品質 | Expectations fail 0件、quarantine が許容範囲内 |

### 4.4 不具合管理（共通）

| 項目 | 定義 |
|------|------|
| 起票必須項目 | 再現手順・期待値・実際値・対象テーブル・対象target |
| 重要度 | 高（データ誤り・処理停止）/ 中（限定的影響）/ 低（軽微） |
| 優先度 | トリアージで決定（重要度 × 業務影響） |
| フロー | 起票 → トリアージ → 修正 → 再試験 → クローズ |

### 4.5 カバレッジ方針（共通）

ロジックカバレッジは pytest（+ coverage）で測定する。SQL/宣言的変換は入力データのケース網羅で実質カバレッジを担保する。目標値は各レベル方針書で定義する。

## 5. テスト環境・データ方針（共通）

| 項目 | 共通方針 | 詳細 |
|------|----------|------|
| 環境分離 | target（dev/staging/prod）単位で catalog/schema を分離 | 各レベル方針書 |
| スキーマ分離 | テスト実行ごとに一意スキーマを生成し干渉を防ぐ | 命名規則は各レベル方針書 |
| テストデータ | 合成データ（fixtures）中心、本番相当は要マスキング | 設計は各レベル方針書 |
| teardown | 失敗時も含めリソースを確実に削除 | パターンは結合テスト方針書 |

> 本番データの非マスキング利用は禁止。

## 6. テスト自動化方針（共通）

| 要素 | 採用技術 | 用途 |
|------|----------|------|
| テストフレームワーク | pytest | 実行、parametrize、marker |
| 資産定義・デプロイ | DAB | targets / 変数 / Job / Pipeline 定義 |
| 実行基盤 | Lakeflow Jobs | テスト用ジョブ実行・パラメータ受け渡し |
| CI/CD | GitHub Actions | PR連動の自動テスト、環境保護ルール |

PR時に単体テストを自動実行、結合テストは {対象ブランチ/承認} 契機で staging 相当環境にて実行する。本番影響のあるデプロイは環境保護ルールで承認必須とする。marker/parametrize の運用詳細は各レベル方針書で定義する。
