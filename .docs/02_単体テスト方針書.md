## 1. 目的・範囲

### 1.1 目的

個々の変換ロジックが設計どおりに動作することを、最小単位で検証する。下流（結合テスト）に欠陥を持ち込まないことを目的とする。

### 1.2 単体テストの対象

| 対象 | 内容 |
|------|------|
| 変換関数 | PySpark / Python の単一関数・モジュール（純粋ロジック） |
| SQL 変換 | 単一テーブルを生成する SELECT / 変換クエリ |
| Expectation ルール | 個々の品質制約（warn/drop/fail）の判定ロジック |
| 単一テーブルのロード（書込前ロジック） | Hub / Link / Satellite の書込直前 DataFrame を生成する変換ロジック（インメモリで完結する範囲） |
| ユーティリティ | ハッシュキー生成、共通変換、パラメータ解決などの部品 |

### 1.3 対象外

複数テーブル/層をまたぐ整合性、E2Eパイプライン実行、AUTO CDC の履歴遷移、PIT/Bridge の参照整合は結合テストで扱う（本書の対象外）。また、Delta MERGE や AUTO CDC・streaming table などストレージ層の振る舞いそのものの検証も、実テーブルを要するため結合テストで扱う。単体ではこれらの「書込直前 DataFrame の正しさ」までを対象とする。

## 2. 単体テストの観点

### 2.1 共通観点（全体方針書 4.1 を単体向けに具体化）

| 観点 | 確認内容 |
|------|----------|
| 正常系 | 代表入力に対し期待どおりの出力（列・値・型） |
| 境界値 | 桁・長さ・日付境界・ゼロ/最大件数 |
| 異常系 | NULL、空文字、型不正、想定外コード値の扱い |
| 重複・一意性 | キー重複時の挙動、重複排除の正しさ |
| 冪等性 | 同一入力の再実行で同一出力 |

### 2.2 Data Vault 構造の単体観点（全体方針書 4.2 を単体向けに具体化）

| 対象 | 単体での確認項目 |
|------|------------------|
| ハッシュキー生成 | 業務キー → ハッシュの規則一致、NULL/区切り文字の正規化、大文字小文字・トリムの扱い |
| Hub ロード | 業務キーの一意化、load date / record source の付与 |
| Satellite ロード | hash diff による変更検出ロジック、属性ハッシュの算出規則 |
| Link ロード | 関連キーの組み立て、ハッシュの一貫性 |

> 参照先Hubの存在確認やSatellite履歴の時系列整合など「複数テーブル前提」の検証、および Delta MERGE / AUTO CDC のストレージ挙動は結合テストで実施する。本節の確認はいずれも書込直前 DataFrame に対する検証であり、インメモリ（`spark.createDataFrame` / 一時ビュー）で完結する。

### 2.3 Expectations の単体観点

| 観点 | 確認内容 |
|------|----------|
| 判定ロジック | 制約条件が境界で正しく真偽判定する |
| アクション区分 | warn / drop / fail の区分が設計どおり |
| 単一ルール単位 | 1ルールに対する合格データ・違反データの分類 |

## 3. テスト設計・データ

### 3.1 設計技法

同値分割・境界値分析・デシジョンテーブルを用いる。1つの変換ロジックに対し、正常／境界／異常の最小代表ケースを定義し、データ駆動（parametrize）で網羅する。

### 3.2 テストデータ方針

| 種別 | 方針 |
|------|------|
| 固定 fixtures | 観点ごとの最小データを関数内/ファイルで定義し、再現性を担保 |
| インラインデータ | 単一関数テストは入力をテストコード内に明示 |
| 本番データ | 単体では原則使用しない（使用時はマスキング必須） |

### 3.3 期待値の持ち方

期待値はテストコード内に明示（ベタ書き）し、実装ロジックを期待値生成に流用しない（テストと実装の癒着を避ける）。

## 4. 実行方式（pytest）

### 4.1 構成・実行環境

| 項目 | 方針 |
|------|------|
| フレームワーク | pytest |
| Spark 実行 | インメモリ実行を基本とする（{Databricks Connect / ローカルSpark}。プロジェクトで確定）。catalog/schema への書き込みは伴わない |
| 実行契機 | ローカル開発時 + PR時の CI 自動実行（全体方針書 6章） |

### 4.2 pytest 運用ルール

| 項目 | 方針 |
|------|------|
| import mode | `importlib` を基本とし、テストモジュール名の衝突を避ける |
| fixture スコープ | SparkSession 等の重い資源は `session` スコープで共有、データは `function` スコープ |
| parametrize | 正常/境界/異常のケースは `@pytest.mark.parametrize` で表現 |
| marker | `@pytest.mark.unit` を付与し、結合テストと実行を分離 |
| 命名 | `test_<対象>_<観点>` で意図が読めるようにする |

### 4.3 スキーマ/リソースの扱い

単体テストは catalog/schema を使用しない。入力は `spark.createDataFrame()` でインメモリに生成し、SQL変換はテーブル参照を一時ビュー（temp view）に差し替えて検証する。これにより高速性・再現性を確保し、スキーマ生成や teardown の管理を不要とする。

実テーブルを要する検証（Delta MERGE、AUTO CDC、streaming table などストレージ層の挙動、隔離スキーマの生成・teardown）は結合テストで実施する（全体方針書 5章）。

## 5. 合格基準・完了基準

> 全体方針書 4.3 の共通定義に対し、単体レベルのしきい値を設定する。

| 基準 | 単体での設定（例。プロジェクトで確定） |
|------|----------------------------------------|
| ケース消化率 | 計画ケースの 100% を実施 |
| テスト成功率 | 100%（不合格は是正・再実行で解消） |
| カバレッジ | 変換ロジックの行カバレッジ {例: 80%} 以上、主要分岐を網羅 |
| 不具合 | 重要度「高」以上の未解決 0 件 |

カバレッジは pytest + coverage で測定する。SQL/宣言的変換でカバレッジ測定が困難な部分は、入力ケースの網羅（分岐・境界の充足）で代替する。
